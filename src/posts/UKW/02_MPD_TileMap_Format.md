# MPD 瓦片地图格式 — 二进制标记体系的设计艺术

## 概述

`Ultimate Knight WindomXP` 的地图场景使用 **MPD** 格式存储瓦片布局。MPD 是一种**纯二进制**的瓦片地图格式，没有 XML 或 JSON 的冗余开销，依靠精心设计的标记体系（`FFFF0001`、`FFFF00XX`、`00 00 80 3F`）来组织数据。

MPD 的设计展示了**标记化二进制格式**的典型思路：用固定魔数标记不同类型的数据段，解析器通过扫描标记而非固定偏移来定位内容。

---

## 1. 文件头部（8 字节，所有地图一致）

| 偏移 | 大小 | 值 | 说明 |
|:----|:-----|:---|:------|
| 0x00 | 4B | `4D 50 44 20` (`"MPD "`) | 魔数 |
| 0x04 | 1B | `4E` (`'N'`) | 版本标志 |
| 0x05 | 2B | `00 00` | 填充 |
| 0x07 | 2B | `C8 00` = 200 | 固定值（所有地图一致） |

仅仅 8 字节的头部，简洁到了极致。

---

## 2. FFFF0001/FFFF00XX 标记体系

MPD 文件的核心是 **`FFFF00XX`** 标记体系，其中 `XX` 表示段类型：

| 标记 | 间距 | 用途 |
|:-----|:------|:------|
| `FFFF0001` | 204 字节固定 | 通用数据块 |
| `FFFF00XX` + 3B → 明文 | 间距 3 | 文本配置段 |
| `FFFF00XX` + 4B → 名字 | 间距 4 | 名字表条目 |

### 2.1 名字表（通过 `FFFF0001` 定位）

名字表条目通过 `FFFF0001` 标记 + 4 字节偏移定位：

```
FFFF0001 [4B gap] [num] 01 00 00 00 [filename.x\0]
```

- `FFFF0001` 标记在所有地图中稳定可靠
- 跳过 4 字节后，即是 `[num] 01 00 00 00 [filename]`

#### mesh-hit 间距自动检测

同一模型有两个变体（mesh 和 hit，如 `map00.x` 和 `map00hit.x`），它们之间的距离是固定的，但**因文件而异**：

| 地图 | mesh-hit 间距 |
|:-----|:-------------|
| Moon | F5 (245) |
| City | F5 (245) |
| City5 | **F9 (249)** |
| In | F9 (249) |

间距不是由地图类型决定，而是**每个文件独立**。解析器需要运行时自动检测：找到第一个 mesh 名字，测量到其 hit 版本的距离，作为本文件的固定步长。

### 2.2 文本配置段（通过 `FFFF00XX` 定位）

在 `FFFF00XX` 标记后间隔 **3 字节** 即可找到明文内容：

```
FFFF00 XX [3B gap] '明文内容...
```

文本内容一直持续到 `00 00 80 3F` 出现为止。示例：

```
FFFF00 30 [3B] 'BgmFileName(0,fuse.ogg);
```

## 3. `00 00 80 3F` 位置标记

`00 00 80 3F`（float 1.0）是 MPD 中最广泛使用的**位置标记**，出现在每个 tile 条目内部。

### 3.1 标记间距矩阵

| 间距 | 含义 |
|:-----|:------|
| **20B** | 4×4 矩阵对角线元素间距（5 floats × 4B） |
| **>20B** | 矩阵 + 渲染文本段 |

### 3.2 各地图标记间距

| 地图 | 标记数 | 间距模式 | 渲染文本 |
|:-----|:-------|:---------|:---------|
| **In** | 4 | `20,20,20` | 无 |
| **Moon** | 4 | `20,20,20` | 极少（4% 条目） |
| **City2/City** | 8 | `20,20,20,30,20,20,20` | 84% 条目有 |
| **City5** | 6~8 | 重复 `20,20,20,211` | 100% 条目有 |
| **City3** | 4/8 | 混合 | 8% 条目有 |
| **City_tac** | 4 | `20,20,20` | 无 |

渲染文本始终位于 `00 00 80 3F` 标记后 **+6 字节**：

```
00 00 80 3F [6B gap] @ShadowFlag=1; @CullFar=80; ...
```

## 4. 条目结构：Type A 和 Type B

通过系统比对 Moon、In、City5、City 等地图，发现所有 tile 条目都以一个固定模式开头：

```
XX 00 00 00 00 00 80 3F
```

其中 `XX` 是**地图特有的标识字节**。

### 4.1 Type A：3F 起始条目（4 标记，16-16-16 间隔）

```
偏移 0:  [XX]              ← 地图标识字节
偏移 1:  [00 00 00]        ← 固定填充
偏移 4:  [00 00 80 3F]     ← 标记1
偏移 8:  [16B data]        ← 通常为全零
偏移 24: [00 00 80 3F]     ← 标记2
偏移 28: [16B data]        ← 通常为全零
偏移 44: [00 00 80 3F]     ← 标记3
偏移 48: [16B: 坐标区]     ← X, Y(=0), Z
偏移 64: [00 00 80 3F]     ← 标记4
偏移 68: [index:uint16]    ← tile 索引
偏移 70: [param:uint16]    ← 参数
偏移 72+: [text...]        ← 渲染参数明文（可选）
```

**Moon 地图示例：**
```
01 00 00 00 00 00 80 3F  ← 锚点
00 00 00 00 00 00 00 00 ...(全零)  ← gap1
00 00 80 3F              ← m2
00 00 00 00 ...(全零)     ← gap2
00 00 80 3F              ← m3
00 00 00 00 00 C0 A8 44 00 00 00 00 00 C0 A8 44  ← gap3: X=1350.0, Z=1350.0
00 00 80 3F              ← m4
[09 00]                  ← index=9
[01 00]                  ← param=1
```

### 4.2 Type B：BF 起始条目（4 标记，8-8-24 间隔）

City5、City 等地图大量存在，以 `00 00 80 BF`（-1.0）代替 `00 00 80 3F` 作为起始标记：

```
偏移 0:  [00 00 80 BF]     ← 标记1 (BF)
偏移 4:  [8B data]         ← 通常为全零
偏移 12: [00 00 80 3F]     ← 标记2
偏移 16: [8B data]         ← 通常为全零
偏移 24: [00 00 80 3F]     ← 标记3
偏移 28: [24B: 坐标区]     ← 4B零 + 4B分隔符 + 4B零 + X + 4B零 + Z
偏移 52: [00 00 80 3F]     ← 标记4
偏移 56: [index:uint16]    ← tile 索引
偏移 58: [param:uint16]    ← 参数
偏移 60+: [text...]        ← 渲染参数明文
```

### 4.3 各地图条目类型统计

| 地图 | Type A (XX锚点) | Type B (BF) | 合计 |
|:-----|:---------------|:------------|:----|
| Moon | 84 (0x01) | 0 | 84 |
| In | 255 (0x02) | 274 | ~529 |
| City5 | 441 (0x10) | 807 | 1248 |
| City_tac | 686 (0x01) | 4800 | ~5486 |
| City | 267 (0x02) | 3903 | ~4170 |
| City2 | 267 (0x02) | 3903 | ~4170 |
| City3 | 82 (0x01) | — | ~82 |
| City4 | 196 (0x22) | 1 | ~197 |
| In2 | 28 (0x01) | 215 | ~243 |
| Skyland | 48 (0x0e) | 0 | 特殊 |

## 5. 渲染参数文本标记

在 tile 条目内部，`00 00 80 3F` 后跟的明文包含每条 tile 实例的渲染参数：

**Moon 示例：**
```
@ShadowFlag=1;
@CullFar=80;    'Far
@CullMode=1;    '球カリング
```

**City 示例：**
```
@CullFar=50;    'Far
@AlphaTestFlag=1;   'alphaTest
@ShadowFlag=2;      'shadow
@ShadowDrawFlag=0;  'ShadowDraw
'@CullMode=1;       '攻撃設定
```

含义：
- `@ShadowFlag` — 阴影投影开关
- `@CullFar` — 可见距离（单位）
- `@CullMode` — 裁剪模式
- `@AlphaTestFlag` — Alpha 测试开关
- `@ShadowDrawFlag` — 阴影绘制开关

---

## 6. PopInfo 出生点结构

每个 PopInfo 出生点由三个标记分隔的固定结构组成：

```
标记1: 00 00 80 BF  (-1.0)
18B:   [3 floats 坐标 + 标记 ID + 填充]
标记2: 00 00 80 3F  (1.0)
8B:    [2 × uint16 参数 + 文本头]  
文本:  '[N]TeaamNo \r\nPopInfo(N, ...);\r\n'  (变长)
9B:    00 2E BD 3B B3 00 00 00 00  ← 固定 9 字节后缀
标记3: 00 00 80 3F  (1.0)
```

## 总结

MPD 格式的设计亮点：

- **标记化定位**：不依赖固定偏移，通过 `FFFF0001`/`FFFF00XX`/`00 00 80 3F` 三级标记体系定位数据
- **紧凑存储**：每条 tile 仅 70~80 字节，包含索引、坐标、变换矩阵和渲染参数
- **双类型兼容**：Type A（3F 起始）和 Type B（BF 起始）两种条目结构共存，解析器需同时支持
- **地图自适应**：mesh-hit 间距、条目锚点值、标记间距全部由解析器运行时自动检测

这种设计在 2009 年的同人游戏中展示了相当成熟的二进制格式工程能力。
