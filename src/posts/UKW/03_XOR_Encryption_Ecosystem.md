# XOR 加密生态 — 起动战士的 MOD 兼容性设计

## 概述

`Ultimate Knight WindomXP` 的大部分二进制文件（`.hod`、`.ani`、`.mpd`、`.sdt`）都使用 **XOR 对称加密**。开发这没有选择复杂的 AES 或自定义算法，而是用了一个简单的 4 字节 XOR 密钥——但真正有趣的是，**每个 MOD 版本使用了不同的密钥**，形成了一套完整的加密生态。

---

## 1. 加密算法

加密算法极其简单：

```c
#define WINDOM_CIPHER_KEY 0x0B7E7759

// XOR 逐 int (4字节) 处理整个文件
void CipherFile(int key, char* path) {
    // ... 逐 4 字节读取文件
    *(int*)(Buffer + index) ^= key;
    // ...
}
```

- **算法**：逐 4 字节 XOR
- **对称性**：同一操作既加密又解密（XOR 的特性）
- **效率**：O(n)，极快

---

## 2. 各 MOD 版本的 XOR 密钥

| MOD 版本 | XOR Key | 说明 |
|----------|---------|------|
| **WindomXP 原版 / PowerUp Kit** | `0x0B7E7759` | 本次分析的目标版本 |
| SeedMod | `0x95127634` | SEED 高达 MOD |
| MSV_MOD | `0x19870430` | MSV 版 |
| SeedMod 2.0.5 | `0xAC510B91` | SeedMod 升级版 |
| United Mod | `0x13322366` | UNITED 版 |
| Raid Mod | `0xEF452301` | RAID 版 |
| The Epic of War | `0x33333323` | 战争史诗 MOD |
| UN Evo | `0x33322166` | UNITED Evolution |

### 密钥设计观察

| MOD | Key | 特点 |
|:----|:----|:------|
| Epic of War | `0x33333323` | 极其对称，几乎全是 `33` |
| UN Evo | `0x33322166` | 类似 Epic of War，可能有继承关系 |
| 原版 | `0x0B7E7759` | 看起来最随机 |
| SeedMod | `0x95127634` | 接近随机分布 |

SOORD 的密钥显然不是随机生成的，而是**有意识地选择**的——Epic of War 和 UN Evo 的密钥相似性暗示了 MOD 间的继承关系。

---

## 3. 各文件的加密状态

并非所有文件都使用 XOR 加密，游戏的文件系统采用了**分级加密策略**：

| 扩展名 | 加密方式 | 解密后格式 | 说明 |
|--------|---------|-----------|------|
| `.x` | **未加密** | 标准 DirectX XFile v3.03 | PowerUp Kit 已明文，可直接解析 |
| `.hod` | XOR (key=0x0B7E7759) | 二进制（部件名+矩阵） | 需解密后读取 |
| `.ani` | XOR (key=0x0B7E7759) | 二进制（"AN2" + "HD2" 块） | 需解密后读取 |
| `.mpd` | XOR (key=0x0B7E7759) | 二进制（"MPD" 瓦片索引） | 需解密后读取 |
| `.sdt` | XOR (key=0x0B7E7759) | 二进制（选择描述） | 需解密后读取 |
| `.fx` | XOR (key=0x0B7E7759) | 文本 (HLSL) | 需解密后读取 |
| `.spt` (文本) | **无加密** | Shift-JIS 文本 | 明文可直接读取 |
| `.spt.a` | 自定义压缩/加密 | 文本 | 需专用解码器 |

### 加密策略的设计逻辑

- **`.x` 文件不加密**：因为需要被 3DSMax、Maya、Deep Exploration 等第三方工具直接读取
- **`.spt` 文本不加密**：方便 MOD 作者直接编辑角色属性
- **`.spt.a` 使用单独加密**：编译后的二进制状态机，加密强度更高
- **`.hod`/`.ani`/`.mpd` 加密**：游戏引擎独有的二进制格式，防止随意修改

这是一个**务实的加密策略**：对第三方工具需要读取的格式不加密，对游戏私有格式加密。

---

## 4. XOR 的局限性

XOR 加密虽然简单高效，但有已知的弱点：

| 弱点 | 说明 |
|:-----|:------|
| **已知明文攻击** | 如果知道文件头（如 `"HOD"`、`"MPD "`、`"AN2"`），可直接反推密钥 |
| **密钥复用** | 所有文件使用同一密钥，破解一个即可解密全部 |
| **无 IV** | 没有初始化向量，相同的明文块产生相同的密文块 |
| **4 字节周期** | 密钥每 4 字节重复，16 字节以上模式可被统计检测 |

但对于 2009 年的同人游戏来说，XOR 加密已经足够——它的目的是**防止随意修改**，而不是抵御专业逆向。

## 5. 社区工具生态

围绕 XOR 加密形成了一个完整的工具生态：

| 工具 | 功能 |
|:-----|:------|
| **WindomTranscoder** | 通用文件加解密，支持所有 MOD 版本 |
| **WindomXP File Decoder** | SPT/ANI 文件加解密 |
| **Entranscode** | 通用加解密 |
| **[起动战士XP] 文件加密器.exe** | 文件夹批量加解密 |
| **ウィンダムXP 文件加解密程序.exe** | 日文版加解密工具 |

### WindomTranscoder 的 Bug

WindomTranscoder 的源码中存在一个**逻辑反转 bug**：

```c
// HasValidExtension 的返回值语义是反的：
//   返回 TRUE  = 扩展名不在合法列表中（"无效"）
//   返回 FALSE = 扩展名在合法列表中（"有效"）
```

这意味着：
- ☑ 勾选 "Limit By Extensions" → 实际处理的是**不合法**扩展名的文件
- ☐ 不勾选 → 正确处理所有文件

这是一个典型的"函数命名与返回值语义相反"的编程错误，在社区中一直存在。

## 6. 多 MOD 兼容性设计

XOR 加密的一个巧妙之处在于——**解析器完全不需要知道密钥**。加密只影响文件读取，不影响解析逻辑：

```
输入资产目录（Robo/ map/ data/）
    │
    ├── .x 文件 → 标准 DirectX 格式，直接解析
    ├── .hod → XOR 解密 → 骨架树 + 变换矩阵
    ├── .ani → XOR 解密 → 骨骼关键帧
    ├── .mpd → XOR 解密 → 瓦片格网
    ├── .spt → Shift-JIS 文本，直接解析
    └── .dds/.png → 标准图像，直接加载
```

解析器做成 key 可配置的，遇到不同 MOD 的文件只需换 key，解析逻辑完全复用。这种设计使得跨 MOD 兼容变得极其简单——一个工具，8 个 key，覆盖所有版本。

---

## 总结

起动战士的 XOR 加密生态展示了：

- **务实主义**：对第三方工具需要的格式不加密，对私有格式加密
- **生态意识**：预设了多 MOD 并存的可能性，只用 key 切换即可兼容
- **社区活力**：围绕加密解密形成了完整的工具链，甚至有 bug 成为社区共识
- **简单即美**：4 字节 XOR 虽然加密强度不高，但对于同人游戏来说正好——足够防止随意修改，又足够简单让 MOD 作者可以逆向

PS：动画文件（.ani）的加密状态有些特殊——PowerUp Kit 版的 ANI 格式比原始版更复杂，部分社区工具尚未完全支持，需要专门的解码流程。
